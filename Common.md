# HTTP и HTTPS
## 1. Что такое **HTTP**?

### Определение
**HTTP (HyperText Transfer Protocol)** — протокол прикладного уровня для передачи гипертекста (веб-страниц, API, файлы).

> **RFC 9110** (2022) — актуальная спецификация.

---

### Основные характеристики

| Характеристика | Описание |
|---------------|--------|
| **Request-Response** | Клиент → запрос, Сервер → ответ |
| **Stateless** | Каждый запрос независим |
| **Текстовой** | Заголовки и тело — читаемые |
| **Порт** | `80` |
| **Транспорт** | TCP |

---

## 2. Структура HTTP-сообщения

### **Запрос (Request)**

```http
GET /users/123 HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0
Accept: application/json
Authorization: Bearer abc123

### JSON-RPC (JRPC)
- **Что это:** Протокол удаленного вызова процедур (Remote Procedure Call, RPC), использующий JSON для кодирования сообщений.
- **Основные особенности:**
  - Легковесный.
  - Транспорт-независимый (HTTP, WebSocket и др.).
  - Поддерживает синхронные и асинхронные вызовы.
- **Формат запроса:**
```json
{
  "jsonrpc": "2.0",
  "method": "subtract",
  "params": [42, 23],
  "id": 1
}
```

Типы сообщений:
Request — клиент вызывает метод.
</br>Notification — запрос без ожидания ответа (id: null).
</br>Batch — массив запросов.
</br>Error — объект с code, message, data.



## REST

### Определение и история
REST (Representational State Transfer) — архитектурный стиль для веб-API, предложенный Роем Филдингом в 2000 году в диссертации. Не протокол, а набор принципов для stateless коммуникации через HTTP.

### Структура и ключевые особенности

Ресурсы: Всё — ресурсы (URI, e.g., /users/123), действия через HTTP-методы (GET, POST, PUT, DELETE).
</br>Форматы: JSON, XML (часто JSON для читаемости).
</br>Stateless: Каждый запрос самодостаточен, без сессий.
</br>HATEOAS: Опционально — ссылки в ответах для навигации.


### Преимущества и недостатки
Плюсы: Масштабируемость, кэширование, простота (стандартные HTTP), SEO-friendly.
</br>Минусы: Over/under-fetching (лишние данные), чатти (много запросов), не для реал-тайм.

```
GET /users/123 HTTP/1.1
Host: api.example.com
Accept: application/json
```

### Статус-коды HTTP
| Группа | Примеры | Описание |
|--------|---------|----------|
| 1xx    | 101 Switching Protocols | Информационные |
| 2xx    | 200 OK, 201 Created, 204 No Content | Успех |
| 3xx    | 301 Moved Permanently, 304 Not Modified | Перенаправление |
| 4xx    | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found | Ошибка клиента |
| 5xx    | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable | Ошибка сервера |


## GraphQL

GraphQL — язык запросов и runtime для API, разработанный Facebook в 2012 году, открыт в 2015. Позволяет клиентам запрашивать точные данные, минимизируя over-fetching.

Схема: Типы (Query, Mutation, Subscription), резолверы для логики.
</br>Запросы: Queries (чтение), Mutations (изменения), Subscriptions (реал-тайм).
</br>Формат: JSON для запросов/ответов.
</br>Один эндпоинт: /graphql.

request: 
```
query { user(id: 123) { name email } }
```

response:
```
{"data":{"user":{"name":"John","email":"john@example.com"}}}
```

## tRPC

tRPC — фреймворк для end-to-end типобезопасных API в TypeScript, созданный в 2020 году. Упрощает RPC без схем, интегрируясь с React/Next.js.

### Структура и ключевые особенности

Сериализация: JSON.
</br>Типы вызовов: Procedures (функции), routers для модульности.
</br>Типизация: Автоматическая (TS-инференс), без code gen.

### Преимущества и недостатки

Плюсы: Типобезопасность без схем, DX (как локальные функции), лёгкий.
</br>Минусы: Только TS, не для polyglot, менее зрелый.

```
export const appRouter = router({ hello: publicProcedure.query(() => 'Hello!') });
```

 транспортные протоколы, которые используются для передачи данных в сетях

## 1. Что такое **транспортный уровень**?

| Уровень OSI | Название | Протоколы |
|------------|---------|----------|
| 4 | **Транспортный** | TCP, UDP, SCTP |
| 3 | Сетевой | IP |
| 2 | Канальный | Ethernet, Wi-Fi |
| 1 | Физический | Кабель, волокно |

> **Транспортный уровень** отвечает за доставку данных между приложениями (через порты).

---

## 2. TCP (Transmission Control Protocol)

### Определение
**TCP** — надёжный, потоковый, ориентированный на соединение протокол.

> RFC 793 (1981) — основа интернета.

---

### Ключевые характеристики TCP

| Характеристика | Описание |
|---------------|--------|
| **Надёжность** | Гарантирует доставку всех пакетов |
| **Порядок** | Пакеты приходят в правильном порядке |
| **Контроль потока** | `window size` — не перегружает приёмник |
| **Управление перегрузкой** | Замедляет передачу при потерях |
| **Соединение** | 3-way handshake |
| **Двунаправленный** | Полнодуплекс |

---

### 3-way handshake (установка соединения)

```text
Клиент → Сервер: SYN
Сервер → Клиент: SYN-ACK
Клиент → Сервер: ACK



# WebSocket и Sockets: Подробное объяснение

## 1. Что такое **Socket**? (Общее понятие)

### Определение
**Сокет** — это программный интерфейс для обмена данными между процессами через сеть.  
Сокет — это **конечная точка (endpoint)** сетевого соединения.

> *"Сокет — это как розетка в стене: ты подключаешь кабель (соединение), и по нему течёт ток (данные)."*

---

### Типы сокетов (по протоколу)

| Тип сокета | Протокол | Особенности |
|-----------|----------|-------------|
| **TCP-сокет** | `SOCK_STREAM` | Надёжный, потоковый, с установкой соединения |
| **UDP-сокет** | `SOCK_DATAGRAM` | Ненадёжный, пакетный, без соединения |
| **Unix-сокет** | Локальный | Для общения процессов на одной машине |

---

### Где используется?
- **Backend**: Node.js (`net`), Python (`socket`), C (`socket()`)
- **Серверы**: Redis, PostgreSQL (внутренние сокеты)
- **Низкоуровневые протоколы**: FTP, SMTP, HTTP (до HTTP/2)

---

## 2. Что такое **WebSocket**?

### Определение
**WebSocket** — это **протокол связи поверх TCP**, обеспечивающий **полнодуплексное (двунаправленное)** соединение между браузером и сервером через **одно долгоживущее соединение**.

> RFC 6455 (2011) — официальная спецификация.

---

### Ключевые особенности WebSocket

| Особенность | Описание |
|------------|--------|
| **Полнодуплекс** | Клиент и сервер могут отправлять данные **одновременно** |
| **Долгоживущее соединение** | Одно соединение на всю сессию (в отличие от HTTP-запросов) |
| **Низкая задержка** | Нет overhead на каждый запрос |
| **Поддержка бинарных данных** | `ArrayBuffer`, `Blob` |
| **Работает через HTTP** | Начинается с `HTTP Upgrade` |

---

### Как работает WebSocket (handshake)

```http
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: x3JJHMbDL1EzLkh9GBhXDw==
Sec-WebSocket-Version: 13
```

# CSRF, CORS, Дуплекс, MITM
## 1. **CSRF (Cross-Site Request Forgery)**

### Определение
**CSRF** — атака, при которой злоумышленник **обманом заставляет авторизованного пользователя** выполнить нежелательное действие на сайте (например, перевод денег).

> Пользователь уже **авторизован** (есть куки), и сайт доверяет запросам с этими куками.

---

### Как работает атака?

```html
<!-- Злоумышленник создаёт страницу на evil.com -->
<img src="https://bank.com/transfer?to=attacker&amount=1000">
```

Пользователь зашёл в bank.com → получил session_id в куках.
Открыл evil.com в другой вкладке.
Браузер автоматически отправил кукиbank.com с запросом.
Банк выполнил перевод — без ведома пользователя.

| Метод                 | Как работает |
|------------------------|--------------|
| CSRF-токен             | Сервер генерирует уникальный токен, клиент отправляет его в заголовке/форме |
| SameSite куки          | SameSite=Lax / Strict — куки не отправляются при кросс-доменных запросах |
| Double Submit Cookie   | Токен в куке + в заголовке — сверяются |
| CORS + credentials     | Браузер блокирует кросс-доменные запросы с куками без разрешения |


## CORS (Cross-Origin Resource Sharing)

CORS — механизм, позволяющий серверу разрешать или запрещать кросс-доменные запросы от браузера.


| Заголовок                        | Назначение |
|----------------------------------|-------------|
| Access-Control-Allow-Origin       | Разрешённые домены (* или конкретный) |
| Access-Control-Allow-Credentials  | Разрешить куки (true) |
| Access-Control-Expose-Headers     | Какие заголовки клиент может читать |
| Access-Control-Max-Age            | Кэширование preflight |


## Дуплекс / Полнодуплекс / Полудуплекс

| Тип          | Описание                     | Пример                  |
|---------------|------------------------------|--------------------------|
| Симплекс      | Только в одну сторону         | Телевидение              |
| Полудуплекс   | По очереди                   | Рация (push-to-talk)     |
| Полнодуплекс  | Одновременно в обе стороны    | Телефон, WebSocket       |


| Протокол   | Дуплекс                          |
|-------------|----------------------------------|
| HTTP        | Полудуплекс (запрос → ответ)     |
| WebSocket   | Полнодуплекс                     |
| TCP         | Полнодуплекс (два потока)        |
| UDP         | Полнодуплекс (но без соединения) |


## MITM (Man-in-the-Middle)

MITM — атака, когда злоумышленник перехватывает и может изменять трафик между клиентом и сервером.

| Сценарий              | Как происходит                                 |
|-----------------------|------------------------------------------------|
| Wi-Fi в кафе          | Подмена DNS, ARP-спуфинг                       |
| Прокси (корпоративный)| Перехват HTTPS с установкой своего сертификата |
| Подмена сертификата   | Злоумышленник выдаёт свой сертификат           |


### Как защититься?

| Метод                      | Как работает                                |
|---------------------------:|---------------------------------------------|
| HTTPS (TLS)                | Шифрование + проверка сертификата           |
| HSTS                       | Запрет HTTP                                  |
| Certificate Pinning        | Жёсткая привязка к сертификату               |
| DNS over HTTPS (DoH)       | Шифрование DNS-запросов                      |
