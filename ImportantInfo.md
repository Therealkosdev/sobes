## Типы компонентов

## Server components

До Next.js 13 весь React-код по умолчанию был клиентским — он приходил на браузер, там выполнялся React.hydrate, и только потом страница становилась интерактивной. 
Это создавало проблемы:

+ Много лишнего JavaScript на клиенте

+ Медленная первая загрузка (особенно на мобильных)

+ Нельзя было прямо в компоненте делать fetch к базе данных, читать файлы и т.д. — всё нужно было выносить в API routes

### Server components
Компоненты, которые рендерятся только на сервере (на Node.js или в Vercel Edge Functions). Их код никогда не попадает на клиент. По факту является просто обычной функцией которая 1 раз выполняется. Можно положить в нашу страницу. Отлично для динамического SEO, так как страница сразу отдается сервером. 

Шаг 1 — Сервер начинает рендерить дерево компонентов
Next.js идёт по дереву app/ и видит:

Server Components → рендерит сразу на сервере (async/await работает)

Client Components → ставит «заглушку» и продолжает рендерить остальное

Шаг 2 — Сервер отправляет ответ по частям (streaming!)
Сразу, как только готов кусок — он уходит в браузер.
Формат ответа — это не обычный HTML, а специальный RSC Payload (React Server Component Payload) — бинарный/текстовый поток, похожий на JSON, но оптимизированный.

+ Могут напрямую делать fetch к базе данных, файлам, внутренним API

+ Могут использовать async/await прямо в компоненте

+ Не имеют доступа к useState, useEffect, onClick и другим хукам/событиям

+ Не могут импортировать клиентские компоненты напрямую (но клиентские могут импортировать серверные)

+ Значительно уменьшают размер JS-бандла на клиенте

+ Автоматически кэшируются и дедуплицируются (очень круто работает с fetch)

```tsx
// Это Server Component по умолчанию!
async function getPosts() {
  const res = await fetch('https://api.example.com/posts', {
    cache: 'force-cache', // или 'no-store' для динамических данных
  });
  return res.json();
}

export default async function Page() {
  const posts = await getPosts(); // ← можно прямо тут!

  return (
    <div>
      <h1>Все посты</h1>
      {posts.map((post: any) => (
        <div key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.body}</p>
        </div>
      ))}
    </div>
  );
}
```

+ Гидратируется в браузере и становится полностью интерактивным

## SSR (Server-Side Rendering)

Server-Side Rendering — это рендеринг React-компонентов на сервере в готовый HTML, который сразу отправляется клиенту. Когда браузер запрашивает страницу, сервер выполняет React-код, получает нужные данные, генерирует полноценный HTML и отправляет его. Клиент получает уже видимую страницу, а затем загружается JavaScript, который "оживляет" её через процесс гидратации. (добавление интерактивности в компонент)

Основное преимущество SSR в том, что пользователь видит контент очень быстро (быстрый First Contentful Paint), и поисковые системы могут индексировать страницу, так как контент есть в HTML. Страница также работает даже если JavaScript отключен или ещё загружается.

Из минусов: сервер испытывает большую нагрузку, так как каждый запрос требует выполнения React-кода. Time to Interactive может быть медленнее, потому что нужно дождаться загрузки и выполнения JavaScript для полной интерактивности. Также сложнее настроить эффективное кеширование.

Гидратация — это процесс, когда React на клиенте "подхватывает" уже отрендеренный HTML и добавляет к нему обработчики событий и делает его интерактивным. При этом React проверяет, что его Virtual DOM совпадает с тем, что отрендерил сервер. Если есть несоответствие (mismatch), React выдаст предупреждение и может пересоздать элементы.

Типичная проблема SSR — это как раз mismatch между серверным и клиентским рендером. Например, если на сервере вы использовали Date.now() или Math.random(), на клиенте получится другое значение, и React будет ругаться.

## CSR (Client-Side Rendering)

Client-Side Rendering — полная противоположность SSR. Сервер отдаёт минимальный HTML с пустым div и ссылкой на JavaScript bundle. Весь рендеринг происходит в браузере: JavaScript загружается, выполняется, React создаёт компоненты, делает запросы к API, получает данные и только потом отрисовывает интерфейс.

Это классический подход для SPA (о них ниже). Создаёте приложение с Create React App — получаете CSR по умолчанию.

Преимущества CSR в том, что можно строить очень богатые интерактивные приложения с плавными переходами. Сервер разгружен, ему нужно только отдавать статику и API. После первой загрузки навигация работает мгновенно, без перезагрузок страницы.

Недостатки: первая загрузка медленная, особенно на слабых устройствах или медленном интернете. Пока не загрузится и не выполнится JavaScript, пользователь видит белый экран или загрузчик. SEO требует дополнительных решений, потому что поисковики видят пустую страницу. И конечно, без JavaScript приложение вообще не работает.

В коде это выглядит просто: у вас есть index.html с div id="root", и React монтируется туда через ReactDOM.render. Все данные подгружаются уже после монтирования через useEffect и fetch.

## SSG (Static Site Generation)

Static Site Generation — это когда вы генерируете HTML на этапе сборки проекта (build time), а не во время запроса. Например, в Next.js есть getStaticProps — функция, которая выполняется при сборке, получает данные и создаёт статические HTML-файлы для каждой страницы.

Когда пользователь запрашивает страницу, сервер просто отдаёт готовый HTML-файл. Это невероятно быстро, потому что никаких вычислений не происходит. Можно положить эти файлы на CDN и раздавать их пользователям по всему миру с минимальной задержкой.

SSG идеален для контента, который меняется редко: блоги, документация, маркетинговые сайты, портфолио. Отличное SEO, потому что контент есть в HTML. Дешёвый хостинг, потому что серверу не нужно ничего вычислять.

Проблема в том, что для обновления контента нужно пересобрать сайт. Если у вас 10000 страниц, сборка может занять много времени. Не подходит для динамического контента вроде дашбордов или соцсетей.

Есть компромисс — ISR (Incremental Static Regeneration) в Next.js. Страницы генерируются статически, но могут обновляться в фоне по расписанию. Указываете revalidate: 60, и страница будет регенерироваться не чаще раза в минуту.

## SPA (Single Page Application)

SPA — это архитектурный подход, когда всё приложение работает на одной HTML-странице. Навигация между "страницами" происходит без перезагрузки — JavaScript просто меняет контент.

Это стало возможным благодаря History API браузера. React Router (или другой роутер) перехватывает клики по ссылкам, меняет URL через history.pushState, и рендерит нужный компонент. Браузер думает, что произошла навигация, но на самом деле это всё JavaScript.

Например, у вас есть роуты для главной, about, контактов. Все они — просто React-компоненты. При клике Router размонтирует старый компонент и монтирует новый. Без запросов к серверу, без перезагрузки страницы.

Преимущества очевидны: плавная навигация, быстрые переходы после первой загрузки, богатый UX как в нативных приложениях. Можно добавить Service Worker и работать offline.

Недостатки: большой начальный bundle (весь код приложения загружается сразу), сложности с SEO без дополнительных мер, нужно управлять состоянием между "страницами".

Оптимизация SPA — это code splitting и lazy loading. React.lazy позволяет подгружать компоненты по требованию. Вместо одного огромного bundle получаете несколько маленьких, которые загружаются по мере необходимости.

## PWA (Progressive Web App)

PWA — это веб-приложение, которое ведёт себя как нативное. Можно установить на домашний экран, работает offline, отправляет push-уведомления, быстро загружается.

Ключевая технология — Service Worker (Service Worker — это специальный JavaScript-файл, который работает в отдельном потоке браузера (в фоне), независимо от страницы, и даёт суперспособности веб-приложению, особенно когда оно превращается в PWA (Progressive Web App).
Простыми словами:
Service Worker — это твой личный прокси-сервер внутри браузера.
Он стоит между твоим сайтом, пользователем и сетью и может:

Кэшировать файлы → сайт работает офлайн
Делать мгновенную загрузку при повторных визитах
Показывать кастомный офлайн-фолбэк
Отправлять пуш-уведомления
Фоновую синхронизацию
Перехватывать и модифицировать любые сетевые запросы

Жизненный цикл Service Worker (2025)

Registration — ты регистрируешь его один раз
Install — первый запуск, тут обычно кэшируем статику
Activate — старый SW умирает, новый берёт контроль
Fetch — перехват всех запросов (главная магия)
Idle / Terminated — браузер может убить его при нехватке памяти). Это JavaScript, который работает в фоне, отдельно от основного потока. Service Worker может перехватывать сетевые запросы и отдавать закешированные ответы, даже когда интернета нет.

Вы регистрируете Service Worker, он устанавливается и кеширует важные ресурсы: HTML, CSS, JS, картинки. Когда пользователь заходит на сайт без интернета, Service Worker отдаёт контент из кеша. Приложение работает.

Второй важный файл — manifest.json. В нём описываете метаданные приложения: имя, иконки, цвета, режим отображения. Браузер предложит пользователю установить приложение на устройство. После установки оно открывается как отдельное приложение, без адресной строки браузера.

Стратегии кеширования в Service Worker:

Cache First подходит для статических ресурсов. Service Worker сначала ищет в кеше, если не нашёл — идёт в сеть. Быстро, но может показывать устаревший контент.

Network First для API и динамических данных. Сначала пытается загрузить из сети, если не получилось — берёт из кеша. Всегда свежие данные, но работает offline как fallback.

Stale While Revalidate — золотая середина. Отдаёт из кеша сразу (stale), но параллельно обновляет кеш из сети. Пользователь видит контент мгновенно, а при следующем заходе получит свежую версию.

PWA позволяет отправлять push-уведомления через Push API, синхронизировать данные в фоне через Background Sync API, работать offline через кеширование. Это мощный инструмент для создания приложений, которые конкурируют с нативными по UX.

## DOM (Document Object Model)

DOM — это программный интерфейс для работы с HTML-документом. Браузер парсит HTML и строит дерево объектов, где каждый элемент — это узел (node). С этими объектами можно работать через JavaScript.

Структура древовидная. Корень — document. У него дети — html. У html — head и body. У body — ваши div, span и так далее. Каждый узел — это объект со свойствами и методами.

В чистом JavaScript работа с DOM выглядит так: document.getElementById для поиска элемента по id, querySelector для поиска по CSS-селектору, createElement для создания нового элемента, appendChild для вставки в дерево.

Проблема прямой работы с DOM в том, что это медленно. Каждое изменение может вызвать reflow (пересчёт позиций всех элементов) и repaint (перерисовку). Если менять DOM в цикле, браузер может тормозить.

Reflow происходит когда меняете размеры, позицию, добавляете/удаляете элементы. Браузер пересчитывает layout всей страницы. Дорогая операция. Repaint дешевле — это просто перерисовка пикселей, когда меняете цвет или фон.

Оптимизация: батчить изменения, менять стили через классы а не по одному свойству, использовать documentFragment для массовых вставок.

Ещё проблема: работа с DOM императивная. Нужно самому создавать элементы, вставлять, удалять, обновлять. Много кода, легко запутаться. Сложно синхронизировать DOM с данными приложения.

React решает эти проблемы через Virtual DOM и декларативный подход. Вы описываете что хотите видеть, React сам разбирается как это сделать эффективно.

## Virtual DOM

Virtual DOM — это лёгкая копия реального DOM в памяти. Обычные JavaScript-объекты, которые описывают структуру UI. React работает с Virtual DOM вместо реального, потому что операции с JS-объектами в тысячи раз быстрее операций с DOM.

Когда вы пишете JSX, он компилируется в вызовы React.createElement, которые создают Virtual DOM элементы. Это просто объекты с type, props, children.

Процесс работы называется reconciliation. Когда state меняется, React создаёт новое Virtual DOM дерево. Затем сравнивает его с предыдущим (diffing algorithm). Находит отличия и создаёт список изменений (патчей). Эти патчи применяются к реальному DOM одной операцией.

Алгоритм diffing использует эвристики. Первое правило: если тип элемента изменился (div стал span), React удаляет старое дерево полностью и создаёт новое с нуля. State вложенных компонентов теряется.

Второе правило: если тип одинаковый, React сравнивает атрибуты. Если className изменился с "old" на "new", React обновит только этот атрибут, не трогая элемент.

Третье правило: для списков детей React использует keys. Без keys React сравнивает детей по позиции. Если добавить элемент в начало списка без key, React подумает что все элементы изменились и пересоздаст их.

С правильными keys React понимает что элементы просто сдвинулись. Key должен быть стабильным и уникальным. Идеально — id из базы данных. Использовать индекс массива как key плохо, потому что при вставке/удалении индексы меняются.

Virtual DOM позволяет React батчить обновления. Вместо того чтобы менять DOM при каждом setState, React накапливает изменения и применяет их все разом. Одно обращение к DOM вместо десяти.

Современный React использует Fiber архитектуру. Это позволяет прерывать рендеринг и возобновлять его, приоритизировать важные обновления, работать параллельно. Например, анимация может иметь высокий приоритет, а обновление списка в фоне — низкий.

## Прототипы (Prototypes)

Прототипы — это механизм наследования в JavaScript. У каждого объекта есть внутренняя ссылка на другой объект — его прототип. Если свойство не найдено в объекте, поиск идёт в прототипе, затем в прототипе прототипа и так далее. Это цепочка прототипов.

У каждого объекта есть `__proto__` (скрытое свойство, ссылка на прототип). У функций-конструкторов есть prototype — объект, который станет прототипом для экземпляров.

Когда создаёте объект через new, его `__proto__` указывает на prototype функции-конструктора. Методы обычно добавляют в prototype, чтобы не дублировать их для каждого экземпляра.

Например, создаёте конструктор Person. В prototype добавляете метод sayHi. Создаёте экземпляр john через new Person. У john нет собственного метода sayHi, но он есть в john.`__proto__` (то есть Person.prototype). JavaScript находит его там.

Цепочка выглядит так: john -> Person.prototype -> Object.prototype -> null. Когда вызываете john.toString(), метода нет в john, нет в Person.prototype, но есть в Object.prototype.

Наследование реализуется через настройку цепочки. Хотите чтобы Dog наследовал от Animal? Делаете Dog.prototype = Object.create(Animal.prototype). Теперь методы Animal доступны экземплярам Dog.

В современном JavaScript используют классы, но под капотом всё равно прототипы. Класс — это синтаксический сахар. Когда пишете class Dog extends Animal, JavaScript настраивает прототипную цепочку автоматически.

В React классовые компоненты используют наследование. Пишете class MyComponent extends React.Component. Поэтому доступны методы из React.Component: setState, forceUpdate, render. Они в прототипе.

С хуками прототипы не так актуальны, потому что функциональные компоненты не используют this и наследование. Но понимание механизма важно для понимания JavaScript в целом.

## Garbage Collector

Garbage Collector (GC) — автоматическое управление памятью в JavaScript. Движок отслеживает объекты и удаляет те, которые больше не нужны. Вам не нужно вручную освобождать память как в C.

Основной алгоритм — Mark-and-Sweep. GC начинает с корней: глобальные переменные, локальные переменные в стеке вызовов, замыкания. Помечает все достижимые объекты (mark). Затем проходит по памяти и удаляет непомеченные объекты (sweep).

Объект считается достижимым, если к нему есть ссылки из корней или из других достижимых объектов. Создали объект, сохранили в переменную — достижим. Переменная вышла из области видимости или вы присвоили ей null — объект может быть удалён (если нет других ссылок).

Циклические ссылки не проблема для современного GC. Два объекта ссылаются друг на друга, но если к ним нет доступа извне, оба будут удалены.

Утечки памяти в React случаются когда создаёте ссылки, которые не очищаются. Классика — забытые таймеры и подписки. Создаёте setInterval в useEffect без cleanup функции. Компонент размонтируется, но таймер продолжает работать и держит ссылку на state.

Правильно: возвращать cleanup функцию из useEffect, которая вызовет clearInterval. React вызовет её при размонтировании.

То же с подписками на события. Подписались на websocket или eventEmitter в useEffect — должны отписаться в cleanup. Иначе обработчик остаётся в памяти вместе со всеми данными из замыкания.

Замыкания на старые данные — ещё одна проблема. Создаёте обработчик в useEffect с пустым массивом зависимостей. Обработчик захватывает state из момента создания. State обновляется, но обработчик всё ещё видит старое значение. Решение — использовать функциональную форму setState или добавить зависимости.

Ссылки в глобальных объектах — классическая утечка. Создаёте глобальный кеш, складываете туда данные, но никогда не удаляете. Память растёт. Решение — очищать кеш в cleanup, или использовать WeakMap, где ключи могут быть собраны GC.

WeakMap полезна когда нужно ассоциировать данные с объектами. Обычный Map держит сильные ссылки — объекты-ключи не будут удалены. WeakMap держит слабые ссылки — когда объект удаляется, запись из WeakMap удаляется автоматически.

Оптимизация для GC: избегайте создания объектов в рендере. Каждый рендер создаёт новый объект — GC работает чаще. Используйте useMemo для стабильных ссылок.

## CSS — подходы и методологии

Способов стилизовать React-компоненты десятки. Каждый со своими плюсами и минусами.

Inline styles — объект стилей прямо в JSX. Удобно для динамических стилей, полная изоляция. Но нет псевдоклассов (hover), нет медиа-запросов, проблемы с производительностью при большом количестве элементов. Используется редко, в основном для динамических значений вроде transform или цветов из props.

CSS Modules — отдельные CSS файлы, которые компилятор (webpack) преобразует в уникальные имена классов. Пишете .button, в DOM становится .Button_button__xY2zK. Полная изоляция, нет конфликтов имён. Все возможности CSS доступны: псевдоклассы, медиа-запросы, анимации.

Импортируете модуль: import styles from './Button.module.css'. Используете: className={styles.button}. Можете композировать стили через composes — один класс наследует другой.

CSS-in-JS — стили пишутся в JavaScript. Популярные библиотеки: styled-components, emotion. Создаёте компонент с помощью styled.div — получаете React-компонент со встроенными стилями.

Огромное преимущество: динамические стили на основе props. Передаёте primary={true} — фон меняется. Можете использовать JS логику для вычисления стилей. Автоматическое удаление неиспользуемых стилей, критические CSS инлайнятся для SSR.

Недостатки: увеличивается размер bundle, есть runtime overhead (стили генерируются в браузере), сложнее отладка. Styled-components добавляет ~15kb, эмоции меньше.

Utility-First CSS (Tailwind) — набор готовых классов для всего. bg-blue-500 для фона, p-4 для padding, hover:bg-blue-700 для ховера. Очень быстрая разработка, не нужно придумывать имена классов, маленький финальный CSS за счёт переиспользования классов.

Минус: HTML становится многословным, куча классов на элемент. Нужно запомнить сотни утилит. Кастомизация через конфиг. Не всем нравится такой подход.

БЭМ (Блок Элемент Модификатор) — методология именования. Блок это самостоятельный компонент: .button. Элемент это часть блока: .button__icon, .button__text. Модификатор это состояние или вариация: .button--primary, .button--large.

Преимущества БЭМ: понятная структура, нет конфликтов имён (уникальные имена), легко читать код и понимать иерархию. Блоки независимы, можно переиспользовать.

Обычно структурируют так: папка Button содержит Button.jsx, Button.css, возможно отдельные файлы для модификаторов.

SMACSS (Scalable and Modular Architecture) — система категорий для CSS. Base — базовые стили для тегов (body, p, h1). Layout — структура страницы (.l-header, .l-sidebar). Module — переиспользуемые компоненты (.card, .button). State — состояния (.is-active, .is-hidden). Theme — темизация (.theme-dark).

Такая организация помогает структурировать большие проекты.

Выбор подхода зависит от проекта. Для небольших компонентных библиотек хороши CSS Modules. Для дизайн-систем — CSS-in-JS или Tailwind. Для больших команд с дизайнерами — возможно традиционный CSS с БЭМ.

## Деструктуризация

Деструктуризация — это синтаксис для извлечения значений из объектов и массивов в переменные. Очень удобно в React.

Объекты: const { name, age } = user извлекает свойства name и age из объекта user. Можно переименовывать: const { name: userName } = user — создаёт переменную userName со значением user.name.

Можно задать значения по умолчанию: const { name, country = 'USA' } = user. Если country нет в user, используется 'USA'.

Вложенная деструктуризация: const { address: { city } } = user извлекает city из вложенного объекта address.

Rest operator: const { name, ...rest } = user. Переменная rest содержит все остальные свойства объекта.

В React это повсеместно используется для props. Вместо function Component(props) пишем function Component({ title, subtitle, onClick }). Сразу извлекаем нужные props.

Можно комбинировать с rest: function Component({ title, ...rest }). Затем <div {...rest}> передаёт все остальные props в div. Удобно для прокидывания className, id и прочего.

Значения по умолчанию полезны: function Button({ text = 'Click', onClick = () => {} }). Если props не передали, используются дефолты.

Массивы деструктурируются по позиции: const [first, second] = array. Можно пропускать элементы: const [first, , third] = array.

Rest для массивов: const [first, ...rest] = array. Первый элемент в first, остальные в rest.

В React это используется для хуков. useState возвращает массив: const [count, setCount] = useState(0). useReducer: const [state, dispatch] = useReducer(reducer, init).

Деструктуризация делает код чище и читабельнее. Меньше повторений props.name, props.age — просто name, age.

## Portal

Portal — способ рендерить дочерние элементы в DOM-узел вне иерархии родителя. Обычно React рендерит компоненты внутри родительского div. Portal позволяет "телепортировать" элемент в другое место DOM.

Создаёте Portal через createPortal(children, domNode). Первый аргумент — что рендерить, второй — куда.

Классический случай — модальные окна. Модал должен быть поверх всего контента, часто с backdrop, который закрывает по клику. Если рендерить модал внутри компонента, могут быть проблемы с z-index, overflow, positioning.

С Portal рендерите модал в document.body или специальный div в конце body. Модал физически находится вне иерархии, можете свободно управлять позиционированием и z-index.

Важная особенность: события в Portal всплывают по React-дереву, не по DOM-дереву. Если Portal находится внутри div с onClick, клик внутри Portal вызовет обработчик. Это удобно — логика событий следует логике компонентов.

Модал обычно включает управление фокусом (trap focus внутри модала), блокировку скролла body, закрытие по ESC, закрытие по клику на backdrop. Всё это делается в useEffect с cleanup.

Tooltip — ещё один пример. Рендерите tooltip в body, позиционируете абсолютно относительно viewport. Вычисляете позицию триггера через getBoundingClientRect, ставите tooltip рядом.

Portal используется в UI-библиотеках для dropdown, popover, toast notifications — всего что должно быть поверх контента и не ограничиваться родительским overflow.

## Reconciliation

Reconciliation — процесс определения изменений между двумя Virtual DOM деревьями и применение минимального набора изменений к реальному DOM.

Когда state или props меняются, компонент ререндерится. React вызывает render() (или функцию компонента), получает новое Virtual DOM дерево. Затем начинается diffing — сравнение с предыдущим деревом.

Алгоритм использует эвристики для быстроты. Нативный алгоритм сравнения деревьев имеет сложность O(n³). React использует эвристики и достигает O(n).

Первая эвристика: элементы разных типов создают разные деревья. Если div стал span, React удаляет div со всем поддеревом и создаёт span с нуля. Быстро, но state вложенных компонентов теряется.

Вторая эвристика: элементы одного типа — обновление атрибутов. React обновляет только изменённые props. className изменился — обновляет только className, не трогая элемент.

Третья эвристика: использование keys для списков. React сравнивает списки детей. Без keys сравнение идёт по позиции. Добавили элемент в начало — React думает что все элементы изменились.

С keys React понимает какой элемент какому соответствует. Элемент с key="5" сдвинулся с позиции 2 на позицию 1? React просто перемещает DOM-узел. Эффективно.

Key должен быть уникальным среди соседей и стабильным. Использовать индекс массива как key плохо: порядок меняется — keys меняются — React пересоздаёт элементы. Использовать Math.random() ещё хуже — каждый рендер новый key, компоненты пересоздаются с нуля.

Лучший key — id из данных. Если данные приходят из API, там обычно есть уникальный id.

Оптимизация reconciliation: React.memo предотвращает ререндер если props не изменились. Обёртка вокруг компонента, делает shallow comparison пропсов. Если одинаковые — старый результат переиспользуется.

useMemo мемоизирует результат вычислений. Дорогие вычисления в компоненте? Оберните в useMemo с зависимостями. Пересчёт только когда зависимости меняются.

useCallback мемоизирует функции. Проблема: каждый рендер создаётся новая функция, даже если логика та же. Если передаёте функцию в дочерний компонент с React.memo, он будет ререндериться из-за новой ссылки. useCallback возвращает

## bind, call, apply - управление контекстом this

### Проблема контекста this в JavaScript

В JavaScript this - это динамический контекст, который зависит от того, как функция была вызвана, а не где она была объявлена. Это часто приводит к проблемам.

Простой пример проблемы:

```javascript
const user = {
  name: 'John',
  greet() {
    console.log(`Hello, ${this.name}`);
  }
};

user.greet(); // "Hello, John" - работает

const greetFunction = user.greet;
greetFunction(); // "Hello, undefined" - this потерян!
```

Когда мы вызываем user.greet(), this указывает на user. Но когда мы сохраняем метод в переменную и вызываем её, this становится undefined (в strict mode) или window (без strict mode).

### call() - вызов функции с явным this

call() позволяет вызвать функцию с явно указанным контекстом this и передать аргументы по отдельности.

Синтаксис: функция.call(контекст, аргумент1, аргумент2, ...)

```javascript
function introduce(greeting, punctuation) {
  console.log(`${greeting}, I'm ${this.name}${punctuation}`);
}

const person1 = { name: 'Alice' };
const person2 = { name: 'Bob' };

introduce.call(person1, 'Hello', '!'); // "Hello, I'm Alice!"
introduce.call(person2, 'Hi', '.'); // "Hi, I'm Bob."
```

Здесь мы вызываем одну и ту же функцию introduce, но с разными объектами в качестве this. Первый аргумент call() - это то, чем станет this внутри функции. Остальные аргументы передаются в функцию как есть.

Практическое применение - заимствование методов:

```javascript
const arrayLike = {
  0: 'a',
  1: 'b',
  2: 'c',
  length: 3
};

// У arrayLike нет метода slice, но мы можем одолжить его у массива
const result = Array.prototype.slice.call(arrayLike, 0, 2);
console.log(result); // ['a', 'b']

// Современная альтернатива - Array.from()
const result2 = Array.from(arrayLike);
```

Ещё пример - вызов родительского конструктора:

```javascript
function Animal(name) {
  this.name = name;
  this.type = 'animal';
}

function Dog(name, breed) {
  Animal.call(this, name); // Вызываем Animal с контекстом текущего объекта
  this.breed = breed;
}

const rex = new Dog('Rex', 'Labrador');
console.log(rex.name); // 'Rex'
console.log(rex.type); // 'animal'
console.log(rex.breed); // 'Labrador'
```

### apply() - вызов функции с массивом аргументов

apply() работает точно так же как call(), но аргументы передаются не по отдельности, а одним массивом.

Синтаксис: функция.apply(контекст, [массив аргументов])

```javascript
function introduce(greeting, punctuation) {
  console.log(`${greeting}, I'm ${this.name}${punctuation}`);
}

const person = { name: 'Charlie' };
const args = ['Hey', '...'];

introduce.apply(person, args); // "Hey, I'm Charlie..."
```

Классическое применение apply() - работа с Math функциями:

```javascript
const numbers = [5, 6, 2, 3, 7];

// Найти максимум в массиве
const max = Math.max.apply(null, numbers);
console.log(max); // 7

// Почему null? Потому что Math.max не использует this
// Первый аргумент может быть любым

// Современная альтернатива - spread оператор
const max2 = Math.max(...numbers);
```

Разница между call и apply проста: call принимает аргументы списком, apply принимает массив. До появления spread оператора apply был незаменим для передачи массива аргументов.

### bind() - создание новой функции с фиксированным this

bind() не вызывает функцию сразу. Вместо этого он возвращает новую функцию с навсегда привязанным this.

Синтаксис: const новаяФункция = функция.bind(контекст, аргумент1, ...)

```javascript
const user = {
  name: 'Dave',
  greet() {
    console.log(`Hello, ${this.name}`);
  }
};

const greet = user.greet;
greet(); // "Hello, undefined"

const boundGreet = user.greet.bind(user);
boundGreet(); // "Hello, Dave" - this зафиксирован!

// Можно вызывать сколько угодно раз
setTimeout(boundGreet, 1000); // Через секунду: "Hello, Dave"
```

bind() особенно полезен для обработчиков событий и колбэков:

```javascript
class Counter {
  constructor() {
    this.count = 0;
    // Без bind this будет потерян при вызове из setTimeout
    this.increment = this.increment.bind(this);
  }
  
  increment() {
    this.count++;
    console.log(this.count);
  }
  
  start() {
    setInterval(this.increment, 1000); // this сохранён благодаря bind
  }
}
```

Частичное применение (partial application) - мощная техника с bind():

```javascript
function multiply(a, b) {
  return a * b;
}

// Создаём функцию "удвоение"
const double = multiply.bind(null, 2);
console.log(double(5)); // 10
console.log(double(10)); // 20

// Создаём функцию "утроение"
const triple = multiply.bind(null, 3);
console.log(triple(5)); // 15

// Более сложный пример
function greet(greeting, name, punctuation) {
  return `${greeting} ${name}${punctuation}`;
}

const sayHelloTo = greet.bind(null, 'Hello');
console.log(sayHelloTo('Alice', '!')); // "Hello Alice!"
console.log(sayHelloTo('Bob', '.')); // "Hello Bob."

const sayHelloToAlice = greet.bind(null, 'Hello', 'Alice');
console.log(sayHelloToAlice('!')); // "Hello Alice!"
console.log(sayHelloToAlice('?')); // "Hello Alice?"
```

### Проблема this в React классовых компонентах

В React классовых компонентах проблема this особенно актуальна:

```javascript
class Button extends React.Component {
  constructor(props) {
    super(props);
    this.state = { clicked: false };
  }
  
  handleClick() {
    this.setState({ clicked: true }); // Ошибка! this === undefined
  }
  
  render() {
    return <button onClick={this.handleClick}>Click</button>;
  }
}
```

Проблема в том, что когда React вызывает handleClick, он не знает о контексте класса. Метод вызывается как обычная функция, this теряется.

Решение 1 - bind в constructor (классическое):

```javascript
class Button extends React.Component {
  constructor(props) {
    super(props);
    this.state = { clicked: false };
    this.handleClick = this.handleClick.bind(this); // Привязка
  }
  
  handleClick() {
    this.setState({ clicked: true }); // Теперь работает
  }
  
  render() {
    return <button onClick={this.handleClick}>Click</button>;
  }
}
```

Решение 2 - class field со стрелочной функцией (современное):

```javascript
class Button extends React.Component {
  state = { clicked: false };
  
  handleClick = () => {
    this.setState({ clicked: true }); // Работает благодаря стрелочной функции
  }
  
  render() {
    return <button onClick={this.handleClick}>Click</button>;
  }
}
```

Стрелочные функции не имеют собственного this, они берут его из окружающего контекста. Когда объявляете метод как стрелочную функцию через class field, this автоматически привязывается к экземпляру класса.

Решение 3 - стрелочная функция в render (работает но неоптимально):

```javascript
class Button extends React.Component {
  handleClick() {
    this.setState({ clicked: true });
  }
  
  render() {
    // При каждом рендере создаётся новая функция
    return <button onClick={() => this.handleClick()}>Click</button>;
  }
}
```

Этот способ работает, но создаёт новую функцию при каждом рендере. Если передаёте эту функцию в дочерний компонент с PureComponent или React.memo, он будет ререндериться каждый раз из-за новой ссылки на функцию.

Передача аргументов в обработчики:

```javascript
class TodoList extends React.Component {
  handleDelete(id) {
    console.log('Deleting', id);
  }
  
  render() {
    return (
      <ul>
        {this.props.todos.map(todo => (
          <li key={todo.id}>
            {todo.text}
            {/* Способ 1 - bind */}
            <button onClick={this.handleDelete.bind(this, todo.id)}>
              Delete
            </button>
            
            {/* Способ 2 - стрелочная функция */}
            <button onClick={() => this.handleDelete(todo.id)}>
              Delete
            </button>
          </li>
        ))}
      </ul>
    );
  }
}
```

Оба способа создают новую функцию при каждом рендере каждого элемента списка. Для больших списков это может быть проблемой производительности. Решение - создать отдельный компонент TodoItem и привязать обработчик там один раз.

### В современном React с хуками проблема исчезает

```javascript
function Button() {
  const [clicked, setClicked] = useState(false);
  
  const handleClick = () => {
    setClicked(true); // this не используется вообще
  };
  
  return <button onClick={handleClick}>Click</button>;
}

// Даже так можно
function Button() {
  const [clicked, setClicked] = useState(false);
  
  return (
    <button onClick={() => setClicked(true)}>
      Click
    </button>
  );
}
```

В функциональных компонентах нет this, все данные берутся из замыканий. Это проще и понятнее.

Если нужен useCallback для оптимизации:

```javascript
function TodoList({ todos }) {
  const handleDelete = useCallback((id) => {
    console.log('Deleting', id);
  }, []);
  
  return (
    <ul>
      {todos.map(todo => (
        <TodoItem 
          key={todo.id} 
          todo={todo} 
          onDelete={handleDelete} 
        />
      ))}
    </ul>
  );
}
```

### Полифил bind для понимания

Чтобы понять как работает bind, можно написать свою версию:

```javascript
Function.prototype.myBind = function(context, ...boundArgs) {
  const originalFunction = this;
  
  return function(...callArgs) {
    return originalFunction.apply(context, [...boundArgs, ...callArgs]);
  };
};

// Использование
function greet(greeting, punctuation) {
  return `${greeting} ${this.name}${punctuation}`;
}

const user = { name: 'Eve' };
const boundGreet = greet.myBind(user, 'Hello');
console.log(boundGreet('!')); // "Hello Eve!"
```

Как это работает: myBind сохраняет оригинальную функцию и контекст, возвращает новую функцию, которая при вызове использует apply для установки контекста и объединения аргументов.

### Основные различия

* call - вызывает функцию немедленно с указанным this и аргументами списком

* apply - вызывает функцию немедленно с указанным this и аргументами массивом

* bind - возвращает новую функцию с зафиксированным this, не вызывая её

На собеседовании могут спросить: "В чём разница между call и apply?" Ответ: только в способе передачи аргументов. call принимает аргументы через запятую, apply принимает массив.

Могут попросить реализовать bind. Покажите понимание замыканий и apply.

Могут спросить про this в стрелочных функциях. Ответ: стрелочные функции не имеют собственного this, они используют this из окружающего лексического контекста. bind на стрелочную функцию не действует.

## AST (Abstract Syntax Tree) - абстрактное синтаксическое дерево

### Что такое AST

AST - это древовидное представление структуры исходного кода. Когда JavaScript-движок или транспайлер (Babel) обрабатывает код, он сначала парсит его в AST, затем работает с этим деревом.

AST - это промежуточный формат между текстом кода и выполнением. Каждая конструкция языка представлена как узел дерева с определённым типом и свойствами.

Простой пример. Код:

```javascript
const x = 5 + 3;
```

AST этого кода (упрощённо):

```json
{
  "type": "VariableDeclaration",
  "kind": "const",
  "declarations": [
    {
      "type": "VariableDeclarator",
      "id": {
        "type": "Identifier",
        "name": "x"
      },
      "init": {
        "type": "BinaryExpression",
        "operator": "+",
        "left": {
          "type": "Literal",
          "value": 5
        },
        "right": {
          "type": "Literal",
          "value": 3
        }
      }
    }
  ]
}
```

Видите структуру? VariableDeclaration содержит VariableDeclarator, который содержит Identifier (имя переменной) и BinaryExpression (инициализация). BinaryExpression содержит оператор и два операнда.

Более сложный пример - функция:

```javascript
function add(a, b) {
  return a + b;
}
```

AST:

```json
{
  "type": "FunctionDeclaration",
  "id": {
    "type": "Identifier",
    "name": "add"
  },
  "params": [
    { "type": "Identifier", "name": "a" },
    { "type": "Identifier", "name": "b" }
  ],
  "body": {
    "type": "BlockStatement",
    "body": [
      {
        "type": "ReturnStatement",
        "argument": {
          "type": "BinaryExpression",
          "operator": "+",
          "left": { "type": "Identifier", "name": "a" },
          "right": { "type": "Identifier", "name": "b" }
        }
      }
    ]
  }
}
```

Дерево отражает структуру кода: FunctionDeclaration с именем, параметрами и телом. Тело - BlockStatement, содержащий ReturnStatement с BinaryExpression.

### JSX и AST

JSX - это синтаксическое расширение JavaScript. Браузеры не понимают JSX, поэтому Babel транспилирует его в обычный JavaScript через AST.

Исходный JSX:

```javascript
const element = <div className="box">Hello</div>;
```

Babel парсит это в AST JSX:

```json
{
  "type": "JSXElement",
  "openingElement": {
    "type": "JSXOpeningElement",
    "name": {
      "type": "JSXIdentifier",
      "name": "div"
    },
    "attributes": [
      {
        "type": "JSXAttribute",
        "name": {
          "type": "JSXIdentifier",
          "name": "className"
        },
        "value": {
          "type": "StringLiteral",
          "value": "box"
        }
      }
    ]
  },
  "closingElement": {
    "type": "JSXClosingElement",
    "name": {
      "type": "JSXIdentifier",
      "name": "div"
    }
  },
  "children": [
    {
      "type": "JSXText",
      "value": "Hello"
    }
  ]
}
```

Затем Babel трансформирует AST JSX в AST обычного JavaScript:

```json
{
  "type": "CallExpression",
  "callee": {
    "type": "MemberExpression",
    "object": { "type": "Identifier", "name": "React" },
    "property": { "type": "Identifier", "name": "createElement" }
  },
  "arguments": [
    { "type": "StringLiteral", "value": "div" },
    {
      "type": "ObjectExpression",
      "properties": [
        {
          "type": "ObjectProperty",
          "key": { "type": "Identifier", "name": "className" },
          "value": { "type": "StringLiteral", "value": "box" }
        }
      ]
    },
    { "type": "StringLiteral", "value": "Hello" }
  ]
}
```

Из этого AST генерируется код:

```javascript
const element = React.createElement(
  "div",
  { className: "box" },
  "Hello"
);
```

Весь процесс: JSX текст → JSX AST → JavaScript AST → JavaScript код.

### Где используется AST

Babel - транспиляция современного JavaScript в старый для совместимости. Babel плагины работают с AST. Например, плагин babel-plugin-transform-arrow-functions находит в AST все ArrowFunctionExpression узлы и заменяет их на FunctionExpression.

Исходный код:

```javascript
const add = (a, b) => a + b;
```

Babel плагин:

```javascript
module.exports = function({ types: t }) {
  return {
    visitor: {
      ArrowFunctionExpression(path) {
        // Находим стрелочную функцию в AST
        const body = path.node.body;
        
        // Если тело не BlockStatement, оборачиваем в return
        const blockBody = t.isExpression(body)
          ? t.blockStatement([t.returnStatement(body)])
          : body;
        
        // Заменяем на обычную функцию
        path.replaceWith(
          t.functionExpression(
            null,
            path.node.params,
            blockBody
          )
        );
      }
    }
  };
};
```

Результат:

```javascript
const add = function(a, b) {
  return a + b;
};
```

ESLint - статический анализатор кода. ESLint правила работают с AST. Правило no-console находит все CallExpression где callee это MemberExpression с object "console".

```javascript
module.exports = {
  create(context) {
    return {
      CallExpression(node) {
        // Проверяем что вызывается метод console
        if (
          node.callee.type === 'MemberExpression' &&
          node.callee.object.name === 'console'
        ) {
          context.report({
            node,
            message: 'Unexpected console statement'
          });
        }
      }
    };
  }
};
```

Когда ESLint находит console.log в коде, AST содержит CallExpression с callee MemberExpression, правило срабатывает.

Prettier - форматировщик кода. Prettier полностью игнорирует исходное форматирование. Он парсит код в AST, затем генерирует новый код из AST согласно своим правилам. Поэтому результат всегда одинаковый независимо от исходного стиля.

TypeScript - проверка типов. TypeScript расширяет JavaScript AST информацией о типах. Когда пишете:

```typescript
function add(a: number, b: number): number {
  return a + b;
}
```

TypeScript AST содержит узлы TypeAnnotation, которых нет в обычном JavaScript. Компилятор анализирует AST, проверяет соответствие типов, затем удаляет аннотации типов и генерирует чистый JavaScript.

Webpack и сборщики - анализ зависимостей. Webpack парсит код, находит в AST все ImportDeclaration и CallExpression с require, строит граф зависимостей. Tree-shaking работает на уровне AST: анализируются экспорты и импорты, неиспользуемый код удаляется.

### Работа с AST на практике

Можно использовать библиотеки для работы с AST. Популярные: @babel/parser (парсинг), @babel/traverse (обход), @babel/generator (генерация кода), @babel/types (создание узлов).

Пример - найти все переменные в коде:

```javascript
const parser = require('@babel/parser');
const traverse = require('@babel/traverse').default;

const code = `
  const x = 1;
  let y = 2;
  var z = 3;
  function foo() {
    const inner = 4;
  }
`;

// Парсим код в AST
const ast = parser.parse(code);

// Обходим дерево
const variables = [];

traverse(ast, {
  VariableDeclaration(path) {
    path.node.declarations.forEach(declaration => {
      variables.push({
        name: declaration.id.name,
        kind: path.node.kind
      });
    });
  }
});

console.log(variables);
// [
//   { name: 'x', kind: 'const' },
//   { name: 'y', kind: 'let' },
//   { name: 'z', kind: 'var' },
//   { name: 'inner', kind: 'const' }
// ]
```

Пример - переименовать все вхождения переменной:

```javascript
const generate = require('@babel/generator').default;

const code = `
  const oldName = 1;
  console.log(oldName);
  const result = oldName + 5;
`;

const ast = parser.parse(code);

// Переименовываем все oldName в newName
traverse(ast, {
  Identifier(path) {
    if (path.node.name === 'oldName') {
      path.node.name = 'newName';
    }
  }
});

// Генерируем код из изменённого AST
const output = generate(ast);
console.log(output.code);
// const newName = 1;
// console.log(newName);
// const result = newName + 5;
```

Пример - удалить все console.log:

```javascript
const code = `
  const x = 1;
  console.log(x);
  const y = 2;
  console.log(y);
  console.warn('warning');
`;

const ast = parser.parse(code);

traverse(ast, {
  CallExpression(path) {
    if (
      path.node.callee.type === 'MemberExpression' &&
      path.node.callee.object.name === 'console' &&
      path.node.callee.property.name === 'log'
    ) {
      path.remove();
    }
  }
});

const output = generate(ast);
console.log(output.code);
// const x = 1;
// const y = 2;
// console.warn('warning');
```

### AST и React

React тоже работает с AST-подобными структурами. Когда пишете JSX, он компилируется в React.createElement, который создаёт объекты - React элементы. Это упрощённый AST вашего UI.

```javascript
// JSX
<div className="container">
  <h1>Title</h1>
  <p>Text</p>
</div>

// Компилируется в
React.createElement(
  'div',
  { className: 'container' },
  React.createElement('h1', null, 'Title'),
  React.createElement('p', null, 'Text')
)

// Создаёт объект (React element)
{
  type: 'div',
  props: {
    className: 'container',
    children: [
      {
        type: 'h1',
        props: { children: 'Title' }
      },
      {
        type: 'p',
        props: { children: 'Text' }
      }
    ]
  }
}
```

Этот объект - Virtual DOM узел, дерево таких объектов - Virtual DOM. React сравнивает два дерева Virtual DOM (reconciliation) и определяет минимальные изменения для реального DOM. Это работает похоже на работу с AST: обход дерева, сравнение узлов, трансформация.

### Типичные вопросы на собеседовании

Что такое AST? - Древовидное представление структуры кода, где каждая языковая конструкция это узел определённого типа.

Где используется AST? - В транспайлерах (Babel), линтерах (ESLint), форматтерах (Prettier), компиляторах (TypeScript), сборщиках (Webpack).

Как Babel транспилирует JSX? - Парсит JSX в AST, трансформирует JSX узлы в CallExpression (React.createElement), генерирует код.

Что такое tree-shaking? - Удаление неиспользуемого кода на основе анализа AST: находятся импорты/экспорты, строится граф, неиспользуемые ветки удаляются.

Как ESLint находит ошибки? - Парсит код в AST, правила обходят дерево (visitor pattern), проверяют узлы определённого типа, находят нарушения.

Можете ли написать простой Babel плагин? - Да, плагин это объект с visitor, где ключи - типы узлов AST, значения - функции для их обработки.

### Визуализация AST

Полезный инструмент для понимания AST - https://astexplorer.net. Можно ввести код, выбрать парсер (например @babel/parser), и увидеть AST визуально. Каждый узел можно развернуть, увидеть его тип и свойства. При клике на узел подсвечивается соответствующий код.

Поиграйтесь с разными конструкциями: функции, классы, JSX, стрелочные функции. Увидите как они представлены в AST. Это сильно помогает понять как работают инструменты вроде Babel.

### Заключение

bind, call, apply - фундаментальные инструменты для управления this. В классовых React компонентах без них не обойтись. В современных хуках проблема решается проще через замыкания.

AST - основа всех современных инструментов разработки. Babel, ESLint, Prettier, TypeScript, Webpack - все работают с AST. Понимание AST помогает понять как работают эти инструменты, как писать плагины, как оптимизировать код.

На собеседовании эти темы могут всплыть в разных контекстах: производительность React, оптимизация сборки, работа с кодом, понимание инструментов. Глубокое знание этих концепций выделит вас среди других кандидатов.
